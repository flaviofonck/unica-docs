    1.1 — Estructura del módulo (menú lateral, 5 opciones)¶

    El módulo debe mostrar en el menú lateral exactamente estas 5 opciones de navegación:

    1. Cargar archivo base de Incidencias

    2. Validación de ANS y exclusiones

    3. Generación de archivos de conciliación

    4. Cargue de respuestas de proveedores

    5. Generación de consolidado de conciliación mensual

    Prueba:

    1. Ingresar a Indisponibilidades en el menú principal.

    2. Verificar que el menú lateral muestra las 5 opciones.

    3. Verificar que cada opción navega a su vista correspondiente sin error 404.

    1.2 — Cargar archivo base de Incidencias¶

    Especificaciones técnicas del archivo Remedy¶

    Parámetro Valor
    Nombre de archivo Closed TERCEROS Preview_MM_AAAA.xlsx
    Hoja BASE GENERAL
    Columnas reales 12 (verificado — columnas 10 y 11 tienen nombre duplicado, parsear por índice)
    Volumen ~20.000 a 30.000 registros por mes
    Llave de cruce ID ENLACE (columna índice 9, base 1)

    Mapeo de columnas (por índice)¶

    Índice Campo en archivo Campo en BD (fact_tickets_indisponibilidad)
    1 Incident ID incident_id
    2 Vendor Ticket Number ticket_tercero
    3 Numero DS ds
    4 Incident Reported Date Time fecha_reportada_incidente
    5 Incident Closed Date Time fecha_cerrada_incidente
    6 Customer Company cliente
    7 Customer City ciudad
    8 Customer Site direccion
    9 Classification Service CI clasificacion_servicio
    10 ID ENLACE id_enlace ← llave de cruce
    11 Resolution Category Tier1 categoria_resolucion_1 (por índice)
    12 Resolution Category Tier2 categoria_resolucion_2 (por índice)
    (ext) Assignee Assigned Group grupo_asignado
    (ext) Resolution Category Tier3 motivo_fallo

    Nota de implementación: Las columnas 11 y 12 del archivo real comparten el mismo nombre de header. Se debe parsear obligatoriamente por índice (0-based: índice 10 y 11), no por nombre de columna.

    Endpoint¶

    POST /api/indisponibilidades/cargar
  Body: multipart/form-data  { file: File }
  Response 202: { jobId: string }

GET /api/indisponibilidades/estado-carga/{jobId}
  Response 200: { jobId, estado, progreso, procesados, insertados, duplicados, errores, detalle[] }

    Flujo de carga asíncrona con barra de progreso¶

    1. El usuario selecciona el archivo XLSX y hace clic en Cargar.

    2. El frontend recibe el jobId y muestra la barra de progreso.

    3. El frontend hace polling cada 2 segundos al endpoint GET estado-carga/{jobId}.

    4. Al completar (estado COMPLETADO), se muestra el modal de resultado con:

    5. Total procesado, insertados, duplicados, errores
    6. Tabla con primeras 200 filas del detalle
    7. Botón Descargar CSV completo

    Pruebas:

    1. Cargar archivo Remedy con ~29.154 registros del mes actual.

      • ✅ Barra de progreso aparece inmediatamente.
      • ✅ El progreso avanza de 0 → 100 % sin bloquear la UI.
      • ✅ Al completar, el modal de resultado se abre automáticamente.
      • ✅ Totales: procesados, insertados, errores correctamente calculados.

    2. Bloqueo de carga duplicada (RN-25):

      • Cargar el mismo archivo por segunda vez para el mismo mes.
      • ✅ Sistema rechaza con mensaje: "Ya existe una carga de incidencias para el mes [Mes/Año]."
      • ✅ No se insertan registros duplicados en fact_tickets_indisponibilidad.

    3. Verificar en la tabla fact_tickets_indisponibilidad que:

      • ds cruzó correctamente con dim_configuracion_servicios.
      • mes_reporte se calculó como el mes vencido del archivo.
      • id_configuracion_servicio fue resuelto automáticamente desde el cruce.

    1.3 — Filtros de validación (aplicados durante la carga)¶

    Se aplican 2 filtros secuenciales que determinan el campo estado de cada ticket:

    Filtro 1 — Validación de configuración de servicios¶

    Condición Valor del campo estado asignado
    Numero DS cruza con dim_configuracion_servicios Y Incident Reported Date Time > Fecha Baja "Registro coincidente DS"
    Numero DS no cruza Se pasa al Filtro 2 sin marcar

    Filtro 2 — Validación de exclusiones Remedy¶

    Condición Valor del campo estado asignado Excepción
    Vendor Ticket Number contiene un patrón de dim_exclusiones_remedy "Registro Descartado Remedy-Vendor" Si ya tiene "Registro coincidente DS" → NO sobrescribir
    Vendor Ticket Number NO contiene patrón de exclusión "Registro identificado Remedy-Vendor" Si ya tiene "Registro coincidente DS" → sí sobrescribir con este valor

    Patrones de exclusión predeterminados en dim_exclusiones_remedy:
    FRACTALIA-PDTI · MAXIMO · UNIX · WALTER BRIDGE · YEPDATA

    Pruebas:

    1. Verificar registro cuyo DS está en inventario y fecha es válida → estado = "Registro coincidente DS" → Filtro 2 no sobrescribe si el Vendor Ticket contiene "MAXIMO".

    2. Verificar registro con DS válido y Vendor Ticket contiene "UNIX" → estado final = "Registro identificado Remedy-Vendor" (sobrescribe).

    3. Verificar registro sin DS en inventario y Vendor Ticket contiene "WALTER BRIDGE" → estado = "Registro Descartado Remedy-Vendor".

    4. Verificar registro sin DS en inventario y sin patrón de exclusión → estado = "Registro identificado Remedy-Vendor".

    1.4 — Validación de ANS y exclusiones¶

    Esta sección contiene 3 tablas paramétricas editables:

    Tabla dim_ans_proveedores¶

    Permite configurar los umbrales de disponibilidad comprometidos por proveedor. El CRUD debe estar activo y los cambios deben reflejarse en los cálculos.

    Campos requeridos: id_proveedor, proveedor, pct_ans_minimo, pct_descuento_1, pct_descuento_2, rango_minutos_1, etc. (según HU-07 sección 3.2).

    Tabla dim_motivos_fallas¶

    Lista de motivos por los cuales no aplica ningún descuento por indisponibilidad.

    Regla (RN-24): Si el campo motivo_fallo del ticket está en esta tabla → pct_descuento_final = 0.

    Tabla dim_exclusiones_remedy¶

    Lista de patrones de texto que identifican tickets a descartar.

    Pruebas:

    1. Agregar un nuevo motivo de falla y verificar que al recalcular, ese motivo genera pct_descuento_final = 0.

    2. Agregar un patrón de exclusión nuevo y verificar que al recargar incidencias, los tickets con ese patrón se descartan.

    3. Eliminar un ANS de proveedor y verificar que el sistema genera error controlado al intentar calcular para ese proveedor.

    Informe de Indisponibilidad INFORME_INDISPONIBILIDAD_MM_AAAA¶

    El sistema genera la tabla de informe filtrando solo los tickets con estado:

    • "Registro coincidente DS"
    • "Registro identificado Remedy-Vendor"

    Los campos del informe incluyen: Id Incidente, DS, fechas, cliente, ciudad, indisp_mins, pct_disponibilidad, pct_descuento, pct_final, proveedor, mes_reporte.

    Fórmula % Disponibilidad (RN-22): 

    pct_disponibilidad = (((dias_mes × 1440) - indisp_minutos) / (dias_mes × 1440)) × 100

    Cálculo indisp_mins: suma de minutos entre Fecha Reportada Incidente y Fecha Cerrada Incidente de todos los tickets del DS en el mes.

    Prueba:

    1. Verificar con un DS conocido:
      • Calcular manualmente indisp_mins sumando los tickets del DS.
      • Aplicar fórmula RN-22 y comparar con el valor mostrado en pantalla.
      • ✅ Diferencia ≤ 1 minuto (margen de redondeo).

    1.5 — Generación de archivos de conciliación¶

    Genera un archivo Excel por proveedor que contiene el informe de indisponibilidad de ese período.

    Parámetro Valor
    Nombre del archivo Informe_Indisponibilidad_{NombreProveedor}_{MM}_{AAAA}.xlsx
    Filtro Solo registros del proveedor seleccionado
    Contenido Todos los campos del informe de indisponibilidad

    Pruebas:

    1. Seleccionar un proveedor y hacer clic en Generar archivo de conciliación.

      • ✅ El archivo se descarga con el nombre correcto.
      • ✅ El archivo contiene solo los registros del proveedor seleccionado.
      • ✅ Los campos pct_disponibilidad, pct_descuento, pct_final tienen valores calculados.

    2. Verificar que se puede generar el archivo para cada proveedor activo del mes.

    1.6 — Cargue de respuestas de proveedores¶

    El proveedor devuelve un archivo con su validación de los tickets. El sistema cruza la respuesta con los registros de la tabla fact_tickets_indisponibilidad para actualizar fact_indisponibilidad_mensual_conciliada.

    Prueba:

    1. Cargar el archivo de respuesta de un proveedor.

      • ✅ Los registros con respuesta del proveedor se actualizan en fact_indisponibilidad_mensual_conciliada.
      • ✅ El sistema muestra resumen: registros aceptados / rechazados / nuevos.

    2. Verificar que mes_facturacion se calcula según el esquema de facturación del proveedor (3 esquemas):

      • Vencido: mes = mes del incidente.
      • Anticipado: mes = mes del incidente + 1.
      • Trimestral: mes = próximo cierre trimestral del contrato.

    2.2 — Cargar Ingresos Davox (archivo FISC)¶

    Especificaciones técnicas del archivo FISC¶

    Parámetro Valor
    Formato ZIP (o RAR) que contiene un archivo TXT
    Separador TXT ~ (tilde)
    Columnas 107
    Volumen ~1.034.341 registros / ~774 MB descomprimido · ~30 MB comprimido
    Encoding latin1 (ISO-8859-1)
    Columna CUS columna 47 (base 1) — campo: CODIGO UNICO DE SERVICIO (CUS)
    Filtro obligatorio ESTADO CUS = "ACTIVO" (columna 3)

    Extracción de mes_informe_fisc desde el nombre del archivo¶

    Regex: (ENE|FEB|MAR|ABR|MAY|JUN|JUL|AGO|SEP|OCT|NOV|DIC)_(\d{2})

    Nombre de archivo de muestra Resultado esperado
    FISC_HIST_SC_30_01_SEP_24_060921.zip 2024-09
    FISC_HIST_31_MAR_25.zip 2025-03
    FIS_HIST_28_FEB_25.zip 2025-02
    1_FISC_TOTAL_EMP_30_JUL_25.rar 2025-07

    Endpoint¶

    POST /api/prefactura/ingresos-davox/cargar
  Body: multipart/form-data  { file: File }  (acepta .zip y .rar)
  Response 202: { jobId: string }

GET /api/prefactura/ingresos-davox/estado/{jobId}
  Response 200: { jobId, estado, progreso, procesados, insertados, errores, mensaje }

    Flujo asíncrono¶

    1. El sistema descomprime el ZIP/RAR en memoria.

    2. Lee el TXT con separador ~, encoding latin1.

    3. Extrae mes_informe_fisc del nombre del archivo con el regex.

    4. Filtra solo registros donde ESTADO CUS = "ACTIVO" (columna 3).

    5. Por cada registro, cruza CUS (col 47) con dim_configuracion_servicios campo ds.

    6. Guarda registros válidos en fact_ingresos_davox.

    Pruebas:

    1. Cargar un archivo FISC real (ZIP):

      • ✅ Archivo se descomprime sin error.
      • ✅ Solo registros con ESTADO CUS = ACTIVO se guardan.
      • ✅ mes_informe_fisc extraído correctamente (verificar con 4 nombres de archivo distintos).
      • ✅ Polling muestra progreso en tiempo real (barra de progreso Angular).

    2. Cargar un archivo RAR:

      • ✅ Mismo comportamiento que ZIP (requiere dependencia junrar en pom.xml).

    3. Verificar cruce con inventario:

      • ✅ CUS de fact_ingresos_davox tiene id_configuracion_servicio correctamente resuelto.
      • ✅ Registros sin CUS en inventario quedan con id_configuracion_servicio = null y aparecen en el log de errores.

    4. Verificar columnas guardadas en fact_ingresos_davox:

      • Mínimo: CICLO, NOMBRE CICLO, NEGOCIO, ESTADO CUS, CUS, CARGO FIJO MES, TIPO FACTURACION, PORCENTAJE IVA, mes_informe_fisc, id_configuracion_servicio.

    2.3 — Generación de Prefacturas¶

    Matriz de prerequisitos (verificación previa)¶

    Antes de generar prefacturas, el sistema muestra:

    Componente Último mes liquidado Acción disponible
    Prorrateo Novedades Último mes ejecutado en HU-06 Link → ir a Prorrateo Novedades
    Mantenimientos Último mes con carga en HU-06 Link → ir a Cargar Mantenimientos
    Indisponibilidades conciliadas Último mes con consolidado en HU-07 Link → ir a Consolidado Mensual

    Regla (RN-29): Si las 3 fechas NO son del mismo mes:

    • ✅ El sistema genera una alerta visible.

    • ✅ Solicita doble confirmación del usuario antes de continuar.

    • Si las 3 fechas coinciden: continuar sin alerta.

    Prueba:

    1. Con los 3 componentes del mismo mes → botón Generar Prefacturas activo, sin alerta.

    2. Con Prorrateo de marzo y los demás de abril → verificar que aparece alerta y el botón requiere doble confirmación.

    3. Con solo 1 componente faltante → verificar alerta + doble confirmación.

    Estructura de FACT_PREFACTURA¶

    Los registros de fact_prorrateo_novedades_servicios se trasladan a fact_prefactura. Para registros con novedad = "Recurrente", se aplican:

    • Cruce con fact_indisponibilidad_mensual_conciliada (llaves: id_configuracion_servicio + mes_facturacion) → descuento por indisponibilidad (VALOR_INDISPONIBILIDAD, PCT_DESCUENTO).

    • Cruce con fact_mantenimientos (mismas llaves) → cobro por mantenimientos.

    Campos calculados clave:

    Campo Fórmula
    total_menos_descuento_indisp total_prorrateos - valor_indisponibilidad
    iva (RN-30) total_menos_descuento_indisp × indicador_iva
    total_con_iva total_menos_descuento_indisp + iva
    MRC_COP (RN-28) Si moneda_mrc = "USD" → mrc × trm_del_mes

    Prueba:

    1. Generar prefacturas para el mes de prueba.
      • ✅ fact_prefactura se pobla con todos los servicios activos del mes.
      • ✅ Solo registros novedad = "Recurrente" tienen descuento de indisponibilidad.
      • ✅ El IVA se calcula sobre total_menos_descuento_indisp × indicador_iva (RN-30).
      • ✅ MRC en USD se convierte a COP usando TRM del mes (RN-28).

    2.4 — Vista: Hoja Prefactura (validación ítem a ítem)¶

    El sistema muestra la tabla de prefacturas del mes con capacidad de validar cada ítem.

    Estados posibles por ítem:

    Estado Descripción
    Válido El analista confirma que el registro es correcto
    No Válido El analista rechaza el registro (requiere observación)
    Por Validar Estado inicial, pendiente de revisión

    Navegación por proveedor: El sistema debe permitir navegar entre proveedores con auto-guardado de cambios al cambiar de proveedor.

    Prueba:

    1. Ver la tabla de prefacturas del mes generado.

      • ✅ Todos los ítems inician con estado "Por Validar".
      • ✅ Se puede cambiar el estado a "Válido" o "No Válido" ítem a ítem.
      • ✅ Al navegar a otro proveedor, los cambios del proveedor anterior se guardan automáticamente.

    2. Filtrar por proveedor específico:

      • ✅ Solo se muestran los ítems del proveedor seleccionado.

    2.5 — Vista: Hoja Resumen¶

    Muestra un resumen consolidado con solo los registros en estado "Válido" del mes.

    Columnas mínimas: Proveedor, DS, MRC, MRC COP, % Descuento Indisponibilidad, Valor Indisponibilidad, Mantenimientos, Total Prorrateos, IVA, Total con IVA.

    Prueba:

    1. Marcar varios ítems como "Válido" y pasar a Hoja Resumen.

      • ✅ Solo aparecen los ítems "Válido".
      • ✅ Los totales (IVA, Total con IVA) se calculan sobre la suma de esos ítems.
      • ✅ Los ítems "No Válido" y "Por Validar" no aparecen en el resumen.

    2. Verificar cálculo IVA (RN-30) con un registro conocido: 

      total_menos_descuento = MRC_COP - valor_indisponibilidad
iva_esperado = total_menos_descuento × indicador_iva (1 = 19%, 0 = 0%)

      • ✅ El valor en pantalla coincide con el cálculo manual (diferencia ≤ $1 COP).

    2.6 — Consolidado final del mes + descarga Excel¶

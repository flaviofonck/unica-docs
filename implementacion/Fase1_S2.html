    G1 — CRUD de Usuarios + restricción RN-04¶

    HU-00 §1a, §1b, §1c · Regla RN-04: Solo Gerente/Líder puede crear/modificar/inactivar usuarios

    Verificación requerida¶

    Backend — verificar que el endpoint de creación de usuario exige rol autorizado: 

    POST /api/usuarios         → solo Gerente/Líder (403 si Practicante/Ingeniero/Analista)
PUT  /api/usuarios/{id}    → solo Gerente/Líder
DELETE /api/usuarios/{id}  → solo Gerente/Líder (o inactivar)

    Frontend — verificar que la pantalla /usuarios tiene:

    • Botón "Nuevo Usuario" visible y funcional para Gerente/Líder

    • Botón "Nuevo Usuario" oculto o deshabilitado para los demás perfiles

    • Formulario con campos: Nombre completo, Correo corporativo, Rol, Cargo, Estado

    • Acción de modificación requiere login con perfil autorizado

    • Log de auditoría registra la acción (fecha, usuario que crea/modifica)

    Smoke test G1¶

    1. Login con practicante@unica.co
   → Ir a /usuarios → "Nuevo Usuario" no visible (o clic → 403)
   → Intentar PUT /api/usuarios/1 → 403 Forbidden ✅

2. Login con admin@unica.co (Gerente)
   → Ir a /usuarios → "Nuevo Usuario" visible
   → Crear usuario nuevo: nombre, correo, rol=Practicante, estado=ACTIVO → 201 Created ✅
   → Verificar en log_acceso que se registró la creación

3. Modificar usuario creado:
   → Cambiar estado a INACTIVO → 200 OK ✅
   → Verificar en log_acceso que se registró la modificación

    Criterio de cierre G1¶

    • [ ] Practicante/Ingeniero/Analista → 403 al intentar crear o modificar usuario

    • [ ] Gerente/Líder → 201 al crear usuario

    • [ ] Log de auditoría registra la acción en log_acceso

    • [ ] Campo estado con opciones ACTIVO/INACTIVO funcional en UI

    G3 — Valores demo en dim_clasificacion¶

    HU-05 §4 — Valores canónicos: aeropuerto, datacenter, satelital, starlink, tradicional, otros

    Situación actual¶

    La tabla dim_clasificacion contiene 11 registros:

    • IDs 1-5 → valores demo de V20 (INTERNET DEDICADO, DATOS CORPORATIVOS, ULTIMA MILLA, BACKBONE, ACCESO EMPRESARIAL)

    • IDs 6-11 → valores canónicos de HU-05 (insertados en V22)

    Acción requerida¶

    Consultar con QCS en la reunión del viernes:

    "La tabla DIM_CLASIFICACION tiene 5 valores del seed inicial de desarrollo (INTERNET DEDICADO, DATOS CORPORATIVOS, ULTIMA MILLA, BACKBONE, ACCESO EMPRESARIAL). La HU-05 define 6 valores canónicos (Aeropuerto, Datacenter, Satelital, Starlink, Tradicional, Otros). ¿Los 5 valores del seed deben eliminarse antes de producción, o deben quedarse como opciones adicionales?"

    Si QCS aprueba eliminación — migración lista para S2/S3¶

    -- V28__cleanup_clasificacion_demo.sql (o siguiente versión libre)
DELETE FROM dim_clasificacion WHERE id IN (1, 2, 3, 4, 5);
-- Verificar antes que ningún servicio activo tenga clasificacion de esos IDs

    ⚠️ Antes de ejecutar: SELECT COUNT(*) FROM dim_configuracion_servicios WHERE clasificacion IN ('INTERNET DEDICADO','DATOS CORPORATIVOS','ULTIMA MILLA','BACKBONE','ACCESO EMPRESARIAL');
    Si > 0 → actualizar esos registros primero o coordinar con QCS el valor de reemplazo.

    Criterio de cierre G3¶

    • [ ] Decisión de QCS documentada (mantener o eliminar)

    • [ ] Si se eliminan: migración ejecutada y verificada antes de S3

    G4 — Validar unicidad TRM por mes (RN-12)¶

    HU-05 §5 · Regla RN-12: Solo 1 registro TRM por año+mes

    Verificación requerida¶

    Backend — dim_trm debe tener un UNIQUE constraint en (anio, mes) o una validación en el servicio que rechace un segundo registro para el mismo mes.

    Frontend — el formulario de DIM_TRM debe mostrar un error claro si se intenta insertar un valor duplicado para el mismo mes.

    Smoke test G4¶

    1. Backend up — abrir /inventario → sección DIM_TRM

2. Registrar TRM para abril 2026: anio=2026, mes=4, valor=4.200
   → 201 Created ✅

3. Intentar registrar segundo TRM para abril 2026: anio=2026, mes=4, valor=4.210
   → Debe rechazar con error de unicidad ✅
   → UI muestra mensaje: "Ya existe un TRM para este mes"

4. Registrar TRM para mayo 2026 → Debe funcionar ✅

    Si falta el constraint en BD¶

    -- Agregar constraint de unicidad si no existe
ALTER TABLE dim_trm ADD UNIQUE KEY uk_trm_anio_mes (anio, mes);

    ⚠️ Verificar con: SHOW CREATE TABLE dim_trm; — si ya tiene UNIQUE KEY en (anio, mes), el constraint está presente.

    Criterio de cierre G4¶

    • [ ] Segundo TRM para el mismo mes → error 409 o equivalente

    • [ ] UI muestra mensaje de error descriptivo

    • [ ] UNIQUE constraint confirmado en BD (SHOW CREATE TABLE dim_trm)

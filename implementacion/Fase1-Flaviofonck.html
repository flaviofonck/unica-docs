    TASK-001: Migración V22 — seed permisos iniciales (6 perfiles × 8 pestañas)¶

    Estado: ✅ Completada — 24/03/2026
    Estimado: 1.5h | Prioridad: 🟡 IMPORTANTE
    Contexto: HU-00 §3b/§3c | Brecha: sin seed → usuario nuevo queda sin acceso

    Problema actual:
    AuthController.obtenerModulosVisibles() filtra permisos donde puede_ver=true. Si no existen registros en permisos para un perfil, el JWT no incluye ningún módulo → el usuario no ve ninguna sección en el menú lateral.

    Solución:
    Crear V22__seed_permisos_perfiles.sql que inserte la matriz completa de permisos por perfil.

    Matriz de permisos a implementar:

    Perfil 1.1 Cotización 1.2 Instalaciones 1.3 Migraciones 1.4 Bajas 2.1 Inventario 3.1 Prorrateo 3.2 Indisponibilidades 3.3 Prefacturación
    Gerente ✅ Ver+Ej+Adm ✅ Ver+Ej+Adm ✅ Ver+Ej+Adm ✅ Ver+Ej+Adm ✅ Ver+Ej+Adm ✅ Ver+Ej+Adm ✅ Ver+Ej+Adm ✅ Ver+Ej+Adm

    Analista Conciliación Terceros ✅ Ver ✅ Ver ❌ ❌ ✅ Ver ✅ Ver+Ej ✅ Ver+Ej+Adm ✅ Ver+Ej
    Analista Unidad Carriers ✅ Ver+Ej ✅ Ver+Ej ✅ Ver ✅ Ver ✅ Ver ✅ Ver+Ej ✅ Ver+Ej ✅ Ver+Ej+Adm

    Practicante ✅ Ver+Ej ❌ ❌ ❌ ✅ Ver ❌ ❌ ❌

    ⚠️ Nota: Esta matriz es una propuesta técnica basada en roles de negocio de HU-00 §3a.
    Confirmar con QCS si la distribución exacta de puedeEjecutar y puedeAdministrar por perfil es correcta antes de aplicar en producción.

    Nomenclatura en BD:

    • modulo = nombre de la sección (ej: 'Gestión de Servicios', 'Inventario de Servicios', 'Novedades y Prefactura')

    • seccion = nombre de la pestaña (ej: '1.1 Cotización a Terceros', '2.1 Inventario Servicios Terceros')

    Criterios de aceptación:

    • [x] La migración V22__seed_permisos_perfiles.sql ejecuta sin errores con INSERT IGNORE

    • [x] Al hacer login con un usuario "Practicante", el JWT incluye modulos: ["Gestión de Servicios", "Inventario de Servicios"]

    • [x] Al hacer login con "Analista Conciliación Terceros", el JWT incluye los módulos correspondientes

    • [x] Al hacer login con "Gerente", el JWT incluye los 3 módulos

    • [x] La UI de Perfiles (/perfiles) muestra las 8 filas de permisos para cada perfil

    • [x] INSERT IGNORE garantiza idempotencia (re-ejecutable sin duplicados)

    TASK-002: Migración V23 — agregar campo pais y estado a dim_ciudades¶

    Estado: ✅ Completada — 24/03/2026
    Estimado: 0.5h | Prioridad: 🟡 IMPORTANTE
    Contexto: HU-05 §8 | Brecha: no hay distinción entre código DANE colombiano y código ISO de países extranjeros

    Problema actual:
    dim_ciudades tiene: id, codigo_dane, nombre_ciudad, departamento. Si se registra una ciudad fuera de Colombia, el campo codigo_dane contiene un código ISO/país, pero no hay forma de distinguirlo ni de excluirlo del cálculo de IVA (RN-10).

    Solución:

    ALTER TABLE dim_ciudades
  ADD COLUMN pais VARCHAR(3) NOT NULL DEFAULT 'COL' AFTER departamento,
  ADD COLUMN estado VARCHAR(20) NOT NULL DEFAULT 'ACTIVO' AFTER pais;

   private List<String> obtenerSeccionesVisibles(Usuario usuario) {
       return usuarioPerfilRepository.findByUsuario(usuario).stream()
           .flatMap(up -> permisoRepository.findByPerfil(up.getPerfil()).stream())
           .filter(p -> Boolean.TRUE.equals(p.getPuedeVer()))
           .map(Permiso::getSeccion)
           .filter(s -> s != null && !s.isBlank())
           .distinct()
           .toList();
   }
